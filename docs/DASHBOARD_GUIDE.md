## 🌐 通用操作

### 🧭 顶部导航栏

页面顶部有四个选项卡：

| 选项卡 | 对应页面 | 一句话说明 |
|--------|---------|-----------|
| **记忆** | 记忆页面 | 浏览、创建、编辑记忆 |
| **审查** | 审查页面 | 查看每次修改的变更记录，决定保留还是回滚 |
| **维护** | 维护页面 | 清理不再需要的旧记忆 |
| **观测** | 观测页面 | 查看系统运行状态和搜索性能 |

### 🔑 设置 API 密钥

页面右上角通常会出现下面三种状态之一：

- **设置 API 密钥**：浏览器里还没有存本地 key；点击后会打开**首启配置向导**
- **更新 API 密钥 / 清除密钥**：浏览器里已经存了一把本地 key
- **运行时密钥已启用**：页面拿到了运行时注入的 key

如果浏览器里既没有已保存的 key，也没有运行时注入的 key，且服务端 setup 状态也没显示 Dashboard 鉴权已经可用，首启配置向导可能会在第一次打开页面时自动弹出。

如果 Dashboard 外壳已经打开，但受保护的数据还是没加载出来，最常见的处理方式是：

点击 **设置 API 密钥** → 打开首启配置向导 → 输入你在 `.env` 中配置的 `MCP_API_KEY` 值 → 选择 **只保存 Dashboard 密钥** 或 **保存到本地 `.env`**。

> 如果你在本地调试时设置了 `MCP_API_KEY_ALLOW_INSECURE_LOCAL=true`，并且请求直连 loopback 地址，页面可以不输入密钥就显示受保护数据。

**关于这两个保存选项：**

- **只保存 Dashboard 密钥**：把 key 保存在当前浏览器会话（`sessionStorage`），直到你清除或浏览器会话结束。保存后立即在当前页面生效。
- **保存到本地 `.env`**：仅在“本地 checkout + 非 Docker 运行 + 当前请求直连 loopback”时可用，只会写当前项目里的 `.env*` 文件。Docker 容器场景下此按钮会保持禁用。
- 第一次往本地 `.env` 保存时，Dashboard key 必须非空。
- 向导里的 `Profile C/D` 是按 `router + reranker` 路线设计的预设起点，仍需要你把示例占位值（如 `https://router.example.com/v1`、`router-embedding-model`）替换成真实值，远端 embedding backend 也必须填一个真实的正整数维度。
- provider base 字段填服务 base，不要手动填 `/embeddings`、`/rerank`、`/chat/completions` 这类具体接口后缀；常见后缀会自动去掉，但格式不对或指到 link-local 的地址会被拦下。

### 🌍 语言切换

右上角有一个语言切换按钮，点击可以在中文和英文之间切换。浏览器会自动记住你的选择。

页面初次加载时会在 React 挂载前先按浏览器里保存的语言同步页面标题和 `<html lang>`，刷新后不会先闪回另一种语言的标题。

如果配置向导已经打开，它右上角也有语言切换按钮，切换后已填进去的内容会保留。

> Microsoft Edge 下前端会自动切换到更轻量的视觉模式以减少卡顿；页面结构和操作流程仍然适用。

## 📂 记忆页面

> **这个页面是干什么的？** 浏览和管理你所有的记忆内容。你可以在这里查看已有的记忆、创建新记忆、编辑内容、或者删除不再需要的记忆。

### 🗂️ 页面布局

- **左侧 — 对话记忆仓**：用来创建新的记忆
- **右侧 — 当前节点内容 + 子记忆列表**：查看和编辑已有的记忆

### 🧭 面包屑导航

页面顶部显示当前位置，例如：`根 > core > agent > preferences`。

- 点击路径中任意一级可以直接跳转
- **根** 是最顶层节点，本身不存储记忆内容

### ✍️ 左侧：对话记忆仓

#### 📝 各字段说明

| 字段名 | 界面显示 | 说明 |
|--------|---------|------|
| **记忆标题** | `记忆标题（可选）` | 给这条记忆起一个简短的名字。可以不填，但建议填写。 |
| **对话内容** | 大文本框 | 把你想保存的对话内容粘贴到这里。**必填项**。 |
| **优先级** | `优先级` | 填一个数字。**数字越小，优先级越高**；`0` 比 `5` 更靠前。 |
| **披露条件** | `披露条件` | 控制这条记忆在什么场景下可以被 Agent 自动获取到。例如填 `"仅在用户明确询问时展示"` 或留空表示不限制。 |

> **披露条件**：你可以在这里写一段文字，告诉系统“这条记忆应该在什么条件下被展示给 AI”。如果不确定该填什么，留空即可。
>
> 系统会检查最终路径长度。如果 `父路径 + 标题` 过长，创建会直接报错；改短标题或退回上一层创建即可。

#### ✅ 操作步骤

1. 在 **对话内容** 文本框中粘贴对话
2. （可选）填写 **记忆标题**、**优先级**、**披露条件**
3. 点击 **写入记忆**
4. 系统会先经过 Write Guard 检查
5. 创建成功后，页面底部会显示 `记忆已创建：core://xxx`

> **Write Guard**：写入前的安全检查。Dashboard 这一侧能稳定看到的是“是否继续执行”或返回 `已跳过：write_guard 阻止了 ...`。

### 📄 右侧上方：当前节点内容

显示当前正在查看的记忆的详细内容：

- **内容编辑区**：直接修改记忆的正文
- **优先级 / 披露条件**：修改对应字段
- **保存** 按钮：保存修改
- **删除路径** 按钮：删除这条记忆的访问路径（会先弹确认框）

#### 📌 Gist（摘要 / 原文视图）

每条记忆可能有一个系统自动生成的摘要（Gist）。通过 **摘要** / **原文** 切换按钮在两个视图之间切换。

### 📑 右侧下方：子记忆列表

#### 🔍 子节点筛选

| 筛选项 | 界面显示 | 说明 |
|--------|---------|------|
| **搜索框** | `搜索路径 / 摘要` | 按路径和摘要片段过滤 |
| **最高优先级** | `最高优先级（可选）` | 只显示优先级小于等于该值的记忆 |

- 每张子记忆卡片显示当前优先级、路径/标题和摘要片段
- 点击卡片进入该节点
- 列表底部的 **再加载 N 条** 按钮按批次继续展开

> 如果你正在编辑当前节点的内容，再去点面包屑或子记忆卡片时，页面会先提醒你是否丢弃未保存编辑。

### 🧱 层级视图

记忆页面还有一个 **层级视图** 面板，用来快速看 L0 / L1 / L2 的关系。

- L0 是入口层（根和域）
- L1 是具体记忆节点
- L2 是由多条 L1 记忆派生出来的摘要

你可以展开 L2 摘要查看它的源记忆，以及源内容是否已经和派生时的 hash 不一致。“生成摘要”路径只返回草稿预览，不会因为你打开面板就自动写入数据库。

> 如果面板显示示例 / mock 提示，表示后端没有可展示的真实 L2 数据。

## 📋 审查页面

> **这个页面是干什么的？** 每当记忆被创建、修改或删除时，系统会记录一份“修改前的快照”。这个页面让你看到这些变更，决定保留或回滚。
>
> **作用域**：审查页只显示**当前数据库**对应的快照会话。切换 `.env`、Docker compose project 或 SQLite 文件后，旧库里的会话不会混进来。

### 🗂️ 页面布局

- **左侧 — 审查账本**：历史会话列表和每个会话中的快照
- **右侧 — 差异对比视图**：选中某条快照后展示修改前后的对比

### 📖 左侧：审查账本

#### 🎯 目标会话

顶部下拉框显示 **目标会话**。每次 AI 操作产生的一组修改归入同一个会话。

- 显示 **暂无活跃会话** 表示没有待审查变更
- 选择会话后，下方列出所有修改快照

#### 📸 快照列表

每条快照是一次具体的修改，卡片显示：

- **操作类型**：创建 / 内容修改 / 元数据修改 / 删除 / 别名
- **资源路径**：被修改的记忆路径

点击快照后右侧显示详细对比。

### 🔀 右侧：差异对比

- **红色 / 删除线**：被删除的内容
- **绿色 / 新增**：新增的内容
- **元数据变化**：优先级、披露条件等

删除类快照还会显示这条记忆还能不能被其他路径访问：

- 如果已经没有任何路径能访问它，页面会显示不可逆的孤儿记忆警告
- 如果还有其他路径能访问它，存活路径面板会列出每个 URI，并给每一行一个复制按钮

#### 🎛️ 操作按钮

| 按钮 | 作用 |
|------|------|
| **整合** | 确认这次修改没问题，接受变更并清除快照 |
| **拒绝** | 不接受这次修改，回滚到修改前的状态 |
| **整合全部** | 一次性确认当前会话中所有待审查的变更 |

> 如果你打开快照后，同一路径出现了更晚的内容快照，回滚前会再检查一次。发现内容已变新时会直接返回冲突，不会用旧快照覆盖新内容。
> Metadata-only 回滚也按同一套 fail-close 逻辑：path 在写入前消失返回 `404`；当前 path 指向或 metadata 已变返回 `409`。

## 🔧 维护页面

> **这个页面是干什么的？** 帮你清理不再需要的“垃圾记忆”：过时的、不再被任何路径引用的、活力值衰减很低的。

### 📊 顶部统计卡片

| 卡片 | 含义 |
|------|------|
| **已弃用** | 记忆被更新后留下的旧版本数量 |
| **孤立记忆** | 没有任何路径可以访问到的记忆数量 |
| **低活力** | 当前已加载的低活力清理候选数量。卡片提示会显示其中有多少条可删除。 |
| **遗忘** | 当前已加载的归档候选数量 |

孤儿记忆面板会最先加载。活力和遗忘面板在第一次打开对应 tab 后才加载，所以对应数字会在第一次打开后出现。

### 🧭 维护 tab

维护页现在分成三个 tab：

- **孤儿记忆清理**
- **活力清理**
- **遗忘模拟**

打开某个 tab 时才会加载对应面板。已经打开过的面板会留在页面里，正常切换 tab 不会丢掉 review 状态或当前勾选。

### 🧹 Tab：孤儿记忆清理

> **孤儿记忆**：每条记忆通过一个“路径”来访问。如果路径被删除了，记忆本身可能还在数据库里但无人能访问到——这就叫“孤儿记忆”。

分为两类：

- **已弃用版本**：旧版本记忆
- **孤立记忆**：没有任何路径指向的记忆

#### ✅ 操作步骤

1. 点击 **刷新** 扫描最新的孤儿记忆
2. 勾选你想删除的记忆
3. 点击 **删除 N 条孤儿记忆**
4. 在弹出的确认框中确认删除

> ⚠️ 删除是永久的，无法撤销。
>
> 如果某条 deprecated 记忆仍然是更旧版本链的最终 target，清理弹窗会先拒绝这次删除。

### 💚 Tab：活力清理候选

> **活力值**：每条记忆都有一个活力值，代表它有多“活跃”。新创建的记忆活力值最高，之后会随时间自然衰减。

#### 🔍 筛选条件

| 字段 | 说明 |
|------|------|
| **阈值** | 活力值下限 |
| **非活跃天数** | 多少天没被访问过的记忆才被列入候选 |
| **数量上限** | 最多显示多少条候选，范围 1~500 |
| **域** | 只筛选某个特定域（例如 `core`、`notes`） |
| **路径前缀** | 只筛选路径以某个前缀开头的记忆 |
| **审核人** | 记录是谁发起了这次清理操作；默认 `maintenance_dashboard` |

#### ✅ 操作步骤

1. 填写筛选条件，点击 **应用筛选**
2. 查看候选列表。每条卡片显示：
   - `vitality N.NN`：当前活力值
   - `inactive Nd`：已经多少天没被访问
   - `可删除` 或 `仍有活动路径`
3. 勾选要处理的记忆
4. 选择操作：**准备删除（N）** 或 **准备保留（N）**
5. 点击 **确认 删除** 或 **确认 保留**，输入确认短语
6. 输入正确的确认短语后才会真正执行

**准备删除（N）** 只会把已勾选且标记为“可删除”的候选放进删除 review；仍有活动路径的候选不会被提交删除。

> **执行衰减并刷新**：手动触发一次活力值衰减计算。
>
> 确认失败（短语输错、key 失效、请求超时）时这次准备好的批次会留在页面上，方便你修好问题后重试。

### 🧠 Tab：遗忘模拟与归档候选

维护页还有一个 **遗忘** 面板：

- **衰减模拟**：只读，不会修改数据库
- **候选队列**：低活力记忆，需要人工看一眼
- **保留**：从队列移除，不删除内容
- **归档**：需要先 prepare 再输入确认短语 confirm；后端会把原始记忆标成 deprecated 并写入归档表，不会自动硬删除

> 系统可以帮你找候选，但不能替你自动忘记。任何 archive 写入都需要人工确认。

## 📊 观测页面

> **这个页面是干什么的？** 系统的“体检报告”页面。它显示搜索引擎运行状况、性能数据和后台任务状态。**普通用户日常不需要频繁使用**。

### 📊 顶部统计卡片

| 卡片 | 含义 |
|------|------|
| **查询总数** | 系统总共处理了多少次搜索请求 |
| **延迟** | 搜索请求的平均响应时间（毫秒）。卡片下方小字显示本地化后的 **P95** 延迟 |
| **缓存命中率** | 搜索结果从缓存获取的比例 |
| **索引延迟** | 索引操作的平均耗时 |
| **清理 p95** | 清理操作 95% 的请求在此时间内完成 |
| **清理索引命中率** | 清理操作中索引的命中率 |

### 🎛️ 顶部操作按钮

| 按钮 | 说明 |
|------|------|
| **刷新** | 重新加载所有统计 |
| **重建索引** | 向后端发起一次全量索引重建请求 |
| **睡眠整合** | 向后端发起一次碎片记忆整合请求 |

### 📈 搜索质量面板

观测页的 **搜索质量** 面板展示各检索模式的 MRR、Recall、p95、通道贡献和 RRF 状态。

当前后端有真实鉴权接口，但还没有持久化带标签的质量样本。接口会明确返回 `is_mock=true` / `status=unavailable`，面板上的 MRR / Recall 示例值只用于展示面板长什么样，不能当成真实 benchmark。

> 真实可引用的检索评测以 [EVALUATION.md](EVALUATION.md) 里的公开 benchmark 口径为准。

### 🔎 搜索控制台

让你亲自跑一次搜索来测试系统的检索效果。

#### 📝 各字段说明

| 字段 | 说明 |
|------|------|
| **查询词** | 关键字或问题 |
| **模式** | 混合（推荐）、语义、关键词 |
| **会话 ID** | （可选）填入则优先返回该会话相关结果 |
| **最大结果数** | 最多返回多少条 |
| **候选倍数** | 内部搜索时的候选放大倍数 |
| **优先会话结果** | 勾选后会话内记忆排在前面 |
| **域过滤** | 只在某个特定域中搜索 |
| **路径前缀过滤** | 只搜索某个路径前缀下的记忆 |
| **范围提示** | 给搜索引擎的提示信息 |
| **最大优先级过滤** | 只返回优先级不超过此值的记忆 |

点击 **执行诊断搜索** 后，下方显示结果和诊断信息。

### 🩺 搜索诊断

执行搜索后展示：

- **延迟**：本次搜索花了多少毫秒
- **模式**：实际使用的检索模式
- **交互层级**：快速路径或深度路径
- **意图**：后端返回或采用的意图标签
- **意图 LLM 已尝试**：是否真的调用了 LLM 做意图分类
- **策略**：系统选择的搜索策略
- **降级**：是否被标记为降级；如果发生降级，会显示原因
- **结果列表**：匹配分数、内容摘要、来源路径和更新时间

> 如果某条结果在回包前的路径复核出错，后端会直接把它丢掉并把原因写进“降级”。

### ⚙️ 运行时快照

显示系统当前的运行状态：

- 索引是否正常或降级
- 队列深度
- 最近一次错误
- 睡眠整合状态
- 反思工作流计数（已准备 / 已执行 / 已回滚）

### 📋 索引任务队列

显示正在执行或最近完成的索引任务。每条任务可以：

- **查看**：详细信息
- **取消**：中止正在进行的任务
- **重试**：重新执行失败的任务

## ❓ 常见问题

### Q: 页面打开后什么数据都没有？

通常是因为还没配置 API 密钥。点击右上角 **设置 API 密钥** 打开首启配置向导，输入 `.env` 中的 `MCP_API_KEY`，先用 **只保存 Dashboard 密钥** 让浏览器通过鉴权即可。只有在“本地 checkout + 非 Docker 运行”场景下，才建议顺手用它写 `.env`。

### Q: 点击“写入记忆”后提示“已跳过”？

这说明 Write Guard 阻止了这次写入。Dashboard 不一定能直接看到后端的具体理由；可以修改内容后重试，或查看后端返回结果。

### Q: “优先级”应该填多大的数字？

没有固定范围。建议：非常重要的核心记忆填 `0`，普通记忆填 `5`，弱提示信息用更大的数字。**数字越小越重要**。

### Q: “整合”和“拒绝”操作能撤销吗？

- **整合** 后快照记录会被清除，但记忆内容本身不会变化
- **拒绝** 会把记忆回滚到修改前的状态；如果回滚后又需要这个修改，需要重新创建

### Q: 为什么回滚或编辑保存时返回 `409`？

说明你打开页面以后，这条记忆在别处（MCP tool、另一个浏览器标签页、另一条 Review session）已经被修改了。刷新页面拿到最新内容后重新操作即可。

`404` 则是 path 在你操作前已经被删了。

### Q: 为什么页面突然只剩下一页兜底错误页？

这是 Dashboard 根入口的兜底页：前端 render 阶段撞上了意外错误，先把页面停在最小恢复状态。先直接刷新一次；如果还是回到这页，再把浏览器控制台和后端日志一起留给排查。

### Q: 活力值会自动恢复吗？

不会。活力值只会随时间衰减。但每次记忆被访问（读取或搜索命中）时，活力值会得到一定程度的“刷新”，减缓衰减速度。

### Q: 什么时候需要手动“重建索引”？

当你发现搜索不出明明存在的记忆时，可以尝试重建索引。正常使用情况下索引会自动维护。
